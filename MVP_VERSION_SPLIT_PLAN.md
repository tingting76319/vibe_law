# Vibe Law 系統規劃與 MVP 版本切分 (基於 v0.6.0-beta.1)

## 1. 目標定義 (North Star)
最終目標：
- 在最短時間找到合適辯護律師。
- 同時提供該個案最佳訴訟策略，達到最佳媒合效果。

MVP 三大能力：
1. 系統可理解律師/法官訴訟邏輯與不同法院判斷標準。
2. 當事人上傳訴訟內容後，系統可推薦最有利訴訟律師。
3. 律師上傳訴狀或案件資料後，系統可依受審法官與歷史判例趨勢，生成貼近該律師風格的訴狀草案。

## 2. 目前系統盤點 (Current State)
目前版本：`v0.6.0-beta.1`

已完成：
- Judicial API 契約統一（`status/data/error/meta`）。
- `judicial` 查詢、`changelog`、`auth`、`upload` 背景工作機制。
- RAG 多輪問答與引用來源（基礎）。
- backend 測試/coverage gate 與 CI 基礎流程。

尚缺：
- 法院/法官/律師「可計算特徵」資料模型尚未完整。
- 律師媒合分數引擎與可解釋推薦尚未上線。
- 律師風格對齊的訴狀生成器尚未形成閉環。
- Graph RAG / 記憶層（HippoRAG 概念）尚未產品化。

## 3. 系統架構分層 (對齊你提供的流程)

### A. 資料層
- 判決資料收集、清理、切片、索引。
- 實體抽取：法院、法官、律師、案由、法條、判決結果。
- 關係建模：法院-法官、法官-案由、律師-案件結果。

### B. 推理層
- Hybrid RAG：向量檢索 + Graph RAG 關係推理。
- 記憶層：會話記憶 + 領域知識節點快取 + 預先計算特徵。
- LLM 分析與報告生成（風險評估、策略建議、訴狀草案）。

### C. 產品層
- 個案端：案件風險分析、簡易訴狀、律師媒合。
- 律師端：AI 個案報告、訴訟策略分析、AI 訴狀草案。
- 後台端：資料上傳、品質監控、模型評估與版本治理。

## 4. 版本切分 (Roadmap)

## v0.6.1 (2 週) - 基線穩定版
目標：把現有服務提升為可連續交付基線，避免後續功能開發被平台問題拖垮。

核心交付：
- Upload pipeline 進一步強化（超時重試、錯誤追蹤、可觀測性）。
- DB schema 與 migration 制度固定化（不再手動變更欄位）。
- CI 補齊 smoke + integration 快速檢查。

驗收標準：
- 連續 7 天無重大上傳中斷。
- 主流程 API 錯誤率 < 2%。

主要負責：
- Leader1：里程碑/風險控管
- Coder2：平台穩定與資料庫
- Coder3：CI + 測試閘道

## v0.7.0 (3 週) - 法官/法院邏輯引擎 MVP
目標：達成 MVP 目標 #1（理解法官、法院判斷標準）。

核心交付：
- 法官/法院特徵表（案由分布、量刑/賠償區間、勝敗傾向）。
- `court_profile`、`judge_profile`、`judge_trend` API。
- 判決邏輯分析報告（可解釋：引用案例、法條、趨勢來源）。

驗收標準：
- 指定案由可輸出法院差異分析。
- 報告中引用來源可追溯（案例 ID + 法條）。
- 內部評估：法官趨勢解讀可用率 >= 80%。

主要負責：
- Research1：特徵工程 + Graph RAG 規格
- Coder2：特徵管線與 API
- Coder1：分析頁 UI
- Coder3：指標驗證與回歸測試

## v0.8.0 (4 週) - 律師媒合 MVP
目標：達成 MVP 目標 #2（個案上傳後推薦最有利律師）。

核心交付：
- 律師資料模型（專長、案件紀錄、法院適配度、風格向量）。
- 媒合引擎 v1（規則分 + 相似案例分 + 法院/法官適配分）。
- 個案端流程：上傳案件 -> 風險報告 -> Top-N 律師推薦 + 可解釋理由。

驗收標準：
- 單案推薦時間（P95）< 60 秒。
- 推薦結果可解釋（至少 3 個理由：案由/法院/歷史結果）。
- Top-3 推薦被法務顧問認可率 >= 60%。

主要負責：
- Research1：媒合特徵與評估基準
- Coder2：媒合 API、評分管線
- Coder1：個案端推薦介面
- Coder3：推薦品質離線評估 + E2E

## v0.9.0 (4 週) - 律師策略與訴狀生成 MVP
目標：達成 MVP 目標 #3（律師風格對齊的訴狀/策略輸出）。

核心交付：
- 律師端輸入：訴狀、證據摘要、案件上下文。
- 策略生成器：基於受審法官/法院趨勢，輸出攻防建議、風險點、先後順序。
- 風格對齊訴狀草案：依律師歷史文本風格生成（保留可編修結構）。

驗收標準：
- 律師端報告生成時間（P95）< 90 秒。
- 內容含「依據」區塊（相似案例、法條、法官趨勢）。
- 律師主觀評分（風格貼合度）>= 4.2/5。

主要負責：
- Research1：風格模型與策略 prompt/檢索策略
- Coder2：生成 API + 安全護欄
- Coder1：律師端工作台 UI
- Coder3：品質評估、幻覺/引用驗證

## v1.0.0 (2 週) - MVP GA 上線版
目標：完成 MVP 正式交付與運營就緒。

核心交付：
- 個案端 + 律師端 + 後台端整合發布。
- 監控告警、成本監控、模型版本回滾機制。
- 法務與隱私聲明、操作手冊、對外簡報版報告模板。

驗收標準：
- 發布後 2 週系統穩定（無 P0 事故）。
- 端到端流程成功率 >= 95%。
- 目標指標達成：媒合時間、推薦品質、策略可用性。

主要負責：
- Leader1：GA 發布總控 + Telegram 對外回報
- Coder3：Release gate 與部署
- Coder1/2 + Research1：缺陷收斂與最終性能優化

## 5. MVP 指標儀表板 (必追蹤)
- 媒合效率：`Time-to-Match`（秒）
- 推薦品質：`Top-3 Precision / Acceptance`
- 策略品質：`Lawyer Strategy Score`
- 引用可信度：`Citation Traceability Rate`
- 穩定性：`API Error Rate`、`Upload Job Failure Rate`

## 6. 與代理分工對齊 (執行模型)
- Leader1：每週版本拆分、每日 3 次 Telegram 進度統整、風險升級決策。
- Coder1：個案端/律師端 UI 流程與可用性。
- Coder2：後端 API、資料管線、DB schema。
- Coder3：測試策略、CI/CD、release gate。
- Research1：RAG/Graph RAG、媒合與策略模型、評估基準。

## 7. Release Gate (每版都要過)
- CI 全綠（lint + unit + integration + coverage）。
- API 契約文件同步更新。
- 關鍵流程 E2E 通過（個案上傳、推薦、策略生成）。
- RELEASE/CHANGELOG/版本 tag 完整。
